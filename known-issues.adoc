= Known issues and workarounds for NetApp Data Migrator
:icons: font
:imagesdir: ./media/

[.lead]
NetApp Data Migrator (NDM) has the following known issues.

== Workflows

=== Job Run status can be confusing when errors occur

Some Migration job runs encounter errors and show a Completed or Errored status. This can cause confusion when interpreting the Migration job run status. 

Status definitions:

* Completed: A job run has finished, but might contain errors.

* Errored: A job run failed due to a critical issue.

.Workaround

Verify the job run outcome by checking the job run details for any errors, especially if the status is Completed. Do not rely only on the status label until after you address this issue.

=== Migrations fail for jobs in Inactive state

During data migration, the following error message might appear:

----
Source Path: <source-path> & Destination Path: <destination-path> failed due to Inactive job found. Please reactivate or remove the existing job.
----

This issue occurs when NDM updates the migration paths but can't find an active job associated with them. Recreating the path does not activate the job; you must update it manually.

.Workaround

. Go to (http://<control-plane-IP-address>/jobs-list) in NDM. 
. Manually set the job configuration to *Active*.
. Restart the migration job. 

=== Incorrect file listing during cutover without delta changes

The UI might show files as migrated, even if the source data has not changed. This is a display issue and does not affect actual data integrity. 

.Workaround

. If an incremental sync is not scheduled, manually trigger an incremental migration. 
+
If you previously scheduled incremental sync, you can skip this step because the next sync will run automatically. 

. For your next bulk migration, enable automatic incremental migrations by choosing either *Schedule* or *Cron Expression* at Step 2 (Options-  Incremental Sync Schedule). This option is disabled by default.

. After incremental sync has completed, review the NDM reports:
+
* If NDM reports zero files, your source and destination are in sync. 
* If NDM reports files, re-run the migration until the report shows zero files. Then start cutover.

=== Update to file server protocol version requires page refresh to update the export path list

When you update a file server protocol version, you must perform a page refresh to update the export path list. If you don't, you might cause compatibility issues, delays in fetching the export list, or incomplete updates. This might lead to data inconsistencies, failed synchronization, or errors in accessing the latest export information.  

.Workaround

Refresh the page whenever you update the file server protocol version. 

=== Job paused or stalled for more than 20 minutes

You  might need to intervene when you observe network connectivity issues, issues with source or destination volume stability, or both. The job might be in the Paused or Running state without any visible progress. This might happen if the source or destination services go down, or if the worker service experiences downtime.

.Workaround

. Check the source and destination. 
+
If they are offline, restart to restore connectivity.  

. Check the worker status. 
+
If the worker is offline, use SSH to connect to the VM and run the following command: 
+
[source,cli]
----
systemctl restart datamigrator-worker.service
----
. Reboot the VM: 
+
If issue persists, restart the worker VM. 

== Validation

=== File sizes might differ after migration even if counts match

After data migration completes, the total number of files is correct, but some files might have a different size compared to the original source. This can happen if the network is interrupted or if there are problems with the server during file transfer.

.Workaround

. Review the migration Chain of Custody (COC) report to identify files marked as errored. 
. Re-run the migration until the errors are resolved. 

=== Discovery job shows zero files until control plane is refreshed

When running a discovery job, the Job Run Details page might initially show zero files or directories, even though the Discovery report preview displays the actual results. This occurs because the Control Plane updates every 10 minutes. During this interval, the Job Run Details might temporarily show outdated information, even if the job is marked as Completed. 

.Workaround

* Wait for the dashboard to refresh (up to 10 minutes) the Job Run Details page and display the accurate file and directory counts. 
* For immediate confirmation, refer to the Discovery Report Preview, which includes discovered files and directories. 

== Configuration 

=== Migration precheck displays false Insufficient Space warning

During migration prechecks, you might see the following warning, even if the destination has sufficient space: 

----
Insufficient destination space for selected path. Do you still want to proceed with the migration? 
---- 

This can happen if you skip the discovery step and NDM uses a general command that reads the entire block device size instead of the actual dataset size.  

.Workaround

Run Discovery before a Migration run. This ensures that the disk usage information is available for the pre-check operation. If you still see the warning: 

. Confirm that discovery has completed. 
. Manually verify the destination volume has enough space. 
. If there is sufficient space, you can safely proceed with data migration. 

=== Access permission mismatch when SID mapping is used

NDM might report a `Missing ACE in target` error when SID mapping is used. This error indicates an access permission mismatch between the source and target systems because SID mapping was not performed at the root level.   

.Workaround

Provide the CSV mapping for the SID source and target as shown in the following two scenarios:  

[role="tabbed-block"]
====
.Scenario 1
-- 
Provide the SID in the CSV mapping sheet for users or groups deleted or removed from the source Active Directory, as shown in the following example:

|===
| sid_source | sid_target

| S-1-5-21-2444020195-1862089444-1769087368-1000 
| S-1-5-21-3481156262-2863848796-4292454742-512

|===
--

.Scenario 2
-- 
For active users or groups in Active Directory, provide the usernames or group names in the CSV mapping sheet exclusively in lowercase. Include the domain prefix (domain\username), as shown in the following example: 

|===
| sid_source | sid_target 

| rootdomain\user1 | rootdomain\user2 

|===
--
====

== Reporting 


=== Excel displays incorrect permissions in COC report file

When opening the COC report CSV file in Microsoft Excel, some file or folder permissions might appear as #NAME?, for example, -rwxrwxrwx, instead of the actual values. 

This happens because Excel mistakenly treats certain permission strings (starting with - or =) as formulas, leading to display errors. The CSV file itself is correct, this is only a display issue.  

.Workaround

To view correct file and folder permissions open the CSV file using one of the following applications: 

* Google Sheets 

* Apple Numbers 

* Online CSV viewer 

* Text editor, for example, Notepad++

=== No error message when Discovery job fails due to network issues

If the host or destination server goes down during a Bulk Discovery job, NDM might not show an error message. This can give the impression that the job is still running normally.  

Discovery jobs refresh every 30 seconds. If you notice that the file count, directory count, or data size is not updating, this might indicate a network issue.  

.Workaround

. Check the network connectivity:
+ 
.. Open the worker VM terminal. 
.. Ping the IP address of the destination server.
+
If there is no response, the destination might be unreachable.

. Restore the network interface:

.. Use SSH to connect to the destination server:
+
[source,cli]
----
ssh <destination_IP>
----

.. Find the interface name, for example, eth0:
+
[source,cli]
----
ipconfig
----

.. Bring the network interface back online:
+
[source,cli]
----
ifup <interface_name>
----

. If required, repeat the Steps 1 and 2 for the source server.

=== Unable to switch user on Windows worker

Switching to a different user account on Windows worker might fail due to existing network connections. This can prevent access to the file server. 

.Workaround 

. Remove the previous connection by opening Command Prompt on the Windows worker and running the following commands:
+
[source,cli]
----
net use
----
+
[source,cli]
----
net use <IP address> /delete
----

. Switch to the new user account and access the file server.
